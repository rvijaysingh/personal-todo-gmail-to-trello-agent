# Gmail-to-Trello Agent

An automated agent that monitors Gmail for starred emails and creates a
corresponding Trello card for each one. The card name is generated by a local
LLM (Ollama/qwen3) and falls back to the cleaned subject line if the LLM is
unavailable. After creating the card, the agent removes the star and applies
a "processed" label so your inbox stays tidy. Runs unattended on a schedule
via Windows Task Scheduler.

---

## Prerequisites

- **Python 3.13+**
- **Google Cloud project** with the Gmail API enabled and an OAuth 2.0
  client ID (Desktop app type) — credentials downloaded as `credentials.json`
- **Trello account** with a Power-Up API key and user token
  (https://trello.com/power-ups/admin)
- **Ollama** (optional) — running locally with a model loaded (e.g., `qwen3:8b`).
  If Ollama is unavailable, card names fall back to the cleaned email subject.

## Setup

### 1. Clone and install dependencies

```bash
git clone <repo-url>
cd personal-todo-gmail-to-trello-agent
python -m venv .venv
.venv\Scripts\activate        # Windows
pip install -r requirements.txt
```

### 2. Create the Gmail label

In Gmail Settings → Labels, create a new label named exactly:

```
Agent/Added-To-Trello
```

This nested label will appear under an `Agent` parent in Gmail.

### 3. Find your Trello list ID

Open the Trello board in a browser, append `.json` to the URL, and search
for the list named "Backlog to Triage" (or whichever list you want cards
added to). Copy the `id` field from that list object.

Alternatively, call the Trello API:

```bash
curl "https://api.trello.com/1/boards/{BOARD_ID}/lists?key={KEY}&token={TOKEN}"
```

### 4. Configure credentials

Copy the example files and fill in your values:

```bash
# Global secrets (shared across agents on this machine)
cp config/.env.json.example config/.env.json

# Agent-specific settings
cp config/agent_config.json.example config/agent_config.json
```

Edit `config/.env.json` with your Trello API key/token/board ID, Gmail
OAuth credential paths, and Ollama settings. Edit `config/agent_config.json`
with your Trello list ID. See [docs/config.md](docs/config.md) for a full
field reference.

> **Tip:** To share one `.env.json` across multiple agent repos, store it
> outside any repo and set the `ENV_CONFIG_PATH` environment variable to its
> absolute path.

### 5. Authenticate with Gmail (first run only)

```bash
python src/gmail_client.py --reauth
```

This opens a browser window for the OAuth consent flow. After approving, a
`token.json` is saved at the path you configured in `.env.json`. Subsequent
runs refresh the token automatically.

If the token expires later (e.g., after 6 months of inactivity), re-run the
same command.

## Usage

### Manual run

```bash
python src/orchestrator.py
```

**First run:** limited to starred emails from the last `first_run_lookback_days`
(default 7) to avoid importing your entire starred email history.

**Subsequent runs:** all currently starred emails are processed.

### Scheduled run (Windows Task Scheduler)

1. Open Task Scheduler → Create Basic Task
2. Set the trigger (e.g., daily at 08:00)
3. Action: **Start a program**
   - Program: `C:\path\to\repo\.venv\Scripts\python.exe`
   - Arguments: `src/orchestrator.py`
   - Start in: `C:\path\to\repo`
4. Optional: add `ENV_CONFIG_PATH` to the task's environment variables if
   your `.env.json` lives outside the repo

### Re-authenticate Gmail

```bash
python src/gmail_client.py --reauth
```

### Check Ollama health

```bash
python src/llm_client.py
```

## How It Works

For each starred email (oldest first):

1. Extract subject, sender, date, and body from the Gmail API
2. Generate an actionable card name via Ollama (falls back to cleaned subject)
3. Build a card description: metadata header + full email body (truncated at 16,384 chars)
4. Check the processing ledger for duplicates (prevents re-processing on retry runs)
5. Create the Trello card at the top of the configured list
6. Remove the Gmail star and apply the `Agent/Added-To-Trello` label
7. Record the outcome in `data/emails_processed.db`

Cards created in a batch end up ordered newest-at-top in the Trello list
because each card is posted with `pos="top"` and emails are processed
oldest-first.

## Project Structure

```
├── src/
│   ├── orchestrator.py      # Main entry point and processing loop
│   ├── gmail_client.py      # Gmail API: fetch, unstar, label
│   ├── trello_client.py     # Trello API: create card, validate list
│   ├── card_builder.py      # Card name (LLM/fallback) and description
│   ├── llm_client.py        # Ollama wrapper (urllib, no extra deps)
│   ├── db.py                # SQLite processing ledger
│   ├── config_loader.py     # Config loading and validation
│   └── models.py            # Shared dataclasses
├── prompts/
│   └── card_name.md         # LLM prompt template (committed)
├── config/
│   ├── .env.json.example    # Global config template
│   └── agent_config.json.example  # Agent config template
├── tests/                   # 197 unit tests, all mocked
├── docs/                    # Detailed documentation
│   ├── architecture.md      # System design and pipeline
│   ├── risks.md             # Failure modes and mitigations
│   ├── testing.md           # Test strategy and fixture inventory
│   └── config.md            # Configuration field reference
├── data/                    # SQLite DB (gitignored, auto-created)
└── logs/                    # Rotating log files (gitignored, auto-created)
```
