# Risks and Mitigations

Review this document before adding new modules or debugging unexpected
failures. Update it when new failure modes are discovered in production.

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Gmail app password revoked** — app password manually revoked, 2FA disabled, or Google security event | Very Low | Agent cannot fetch emails; run fails at startup | (1) Startup IMAP auth check (`check_imap_auth`) catches this immediately with a clear error: *"IMAP authentication failed — check gmail_sender and gmail_password in .env.json"*. (2) Agent exits before touching any API. (3) `imaplib.IMAP4.error` (bad credentials) is NOT retried — exits on first attempt. (4) Fix: re-generate an app password in Google Account → Security → App passwords. |
| **Transient network / DNS failure at startup** — machine network not ready when Task Scheduler fires; DNS resolves seconds later | Medium | Agent exits immediately without processing any emails; spurious crash alerts sent | (1) `_retry_startup_check()` in `orchestrator.py` retries both Trello list validation (`TrelloError`) and IMAP connectivity (`OSError`, `socket.timeout`, `ConnectionRefusedError`) up to **3 times** with a **30-second backoff** before giving up. (2) Only `imaplib.IMAP4.error` (bad credentials) bypasses the retry and exits immediately. (3) If all 3 attempts fail, one crash alert is sent and the agent exits — same as before, but transient blips self-heal. |
| **Gmail IMAP rate limits** — high-frequency IMAP connections | Low | Individual emails fail; some cards not created on that run | (1) Configurable `processing_delay_seconds` between emails (default 1s). (2) Failed emails retain their star and are retried on the next run. |
| **Ollama unavailable** — service not running, crashed, or model not loaded | Medium | Card names fall back to cleaned subject line; LLM value lost for that run | (1) Health-check Ollama once at startup; log a WARNING visible at the top of the log if unavailable. (2) Card creation never depends on Ollama — the fallback subject cleaner is always ready. (3) `card_name_source='fallback'` recorded in DB for observability. |
| **Trello API failure** — API down, 4xx/5xx error, list archived or deleted | Low | Email card not created; email remains starred for next run | (1) On any `TrelloError`, record the email as `failed_trello_create` in the DB and **do not** remove the star. (2) Validate the target list ID at startup with retry (see transient network row above); exit immediately if the list is not found after retries. (3) The dedup check on the next run prevents duplicate cards if the first run partially succeeded. |
| **Duplicate Trello cards** — label application fails after card creation; next run re-processes the same email | Medium | User sees two Trello cards for the same email | (1) Primary defence: Gmail query always excludes emails with the processed label (`-label:Agent/Added-To-Trello`). (2) Secondary: `check_duplicate` queries `emails_processed` by `gmail_message_id` (only `status='success'` counts) before any Trello call. |
| **Email body encoding issues** — unusual charset, inline images, heavily nested HTML | Medium | Card description shows garbled or empty body | (1) Prefer the `text/plain` MIME part; recurse into `multipart/*` to find it. (2) For HTML-only emails, strip tags with a simple `HTMLParser` subclass (not a full renderer). (3) If body extraction fails entirely, create the card with only the metadata header and a note: *"[Email body could not be extracted]"*. |
| **Large batch on first run** — user has hundreds of starred emails spanning years | Low | Run takes many minutes; may hit API limits | (1) `first_run_lookback_days` (default 7) caps the initial import. (2) Progress logged every 10 emails so the user can monitor. (3) `processing_delay_seconds` provides a configurable throttle. |
| **Config file missing or malformed** — `.env.json` or `agent_config.json` absent, invalid JSON, or missing field | Low | Agent exits at startup before touching any API | `config_loader.load_config()` validates both files at startup and raises `ConfigError` with the exact missing field name. The orchestrator catches `ConfigError` and exits with a clear message before any API call is made. |
| **Prompt template missing** — `prompts/card_name.md` deleted or moved | Low | Agent exits at startup | `orchestrator._load_prompt_template()` checks file existence and exits with an error naming the expected path. |
| **SQLite corruption** — power loss during write, disk full | Very Low | Processing ledger unreadable; dedup check fails open | `INSERT OR REPLACE` + `conn.commit()` minimise the corruption window. If the DB file becomes corrupt, delete it and re-run — all starred emails will be re-evaluated via their Gmail star state. |
| **Alert delivery failure** — SMTP error, Gmail rate limit, or invalid credentials when sending crash/failure alerts | Very Low | Alert not delivered; failure remains visible only in logs | (1) `agent_shared.alerts.notifier` catches all SMTP errors internally and logs them. (2) The orchestrator wraps every alert call in `try/except` so a failing alert never propagates. (3) The agent continues its normal exit path regardless — alerts are best-effort, not critical path. |
